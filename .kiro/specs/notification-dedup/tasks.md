# Implementation Plan

- [x] 1. 送信済み通知テーブルのDBマイグレーション作成
- [x] 1.1 sent_notificationsテーブルのマイグレーションSQLを作成する
  - イベントID、ギルドID、通知キーの3カラムをテキスト型で定義する
  - 送信日時カラムをtimestamptz型でデフォルトnow()として定義する
  - 3カラムの複合ユニーク制約を設定し、重複レコードをDBレベルで防止する
  - 送信日時カラムにインデックスを作成し、クリーンアップクエリを高速化する
  - _Requirements: 1.1, 1.3_

- [ ] 2. 送信済み通知サービスの実装
- [ ] 2.1 送信済みチェック機能を実装する
  - 指定されたイベントID・ギルドID・通知キーの組み合わせでレコードが存在するかを確認する
  - 既存のServiceResultパターンとclassifySupabaseErrorに従う
  - レコードが見つからない場合はfalseを返す
  - _Requirements: 1.1, 2.1_

- [ ] 2.2 送信済み記録機能を実装する
  - イベントID・ギルドID・通知キー・送信日時をテーブルに挿入する
  - ユニーク制約違反（既に記録済み）の場合は成功として扱う
  - 繰り返しイベントの擬似ID（series:{id}:occ:{日時}形式）をそのまま記録できるようにする
  - _Requirements: 1.1, 1.2, 1.4_

- [ ] 2.3 (P) 古いレコードのクリーンアップ機能を実装する
  - 指定日数以上前の送信済みレコードを一括削除する
  - 削除したレコード件数を返却する
  - _Requirements: 3.1, 3.2_

- [ ] 3. 通知タスクへの重複防止機構の統合
- [ ] 3.1 通知送信前の送信済みチェックを追加する
  - 各通知の送信処理前に、送信済みサービスを使って既送信かを確認する
  - 送信済みの場合はスキップし、デバッグログを出力する
  - DBチェック失敗時はフェイルオープン（送信を実行）とし、重複より未送信を防ぐ
  - 送信成功後に送信済みとしてDBに記録する
  - センチネル通知（__start__）も同じフローで処理する
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 4.1, 4.2, 4.3, 4.4_

- [ ] 3.2 クリーンアップの定期実行を組み込む
  - 通知処理サイクルの最後で、前回から1時間以上経過していればクリーンアップを実行する
  - 保持期間は7日とし、それ以上前のレコードを削除対象にする
  - クリーンアップの成否にかかわらず通知処理は継続する
  - 既存コメント（notify.ts L271-275の重複許容コメント）を削除する
  - _Requirements: 3.1, 3.2, 3.3_

- [ ] 4. テストの作成
- [ ] 4.1 (P) 送信済み通知サービスのユニットテストを作成する
  - 送信済みチェック機能の真偽判定をテストする
  - 送信済み記録機能の正常挿入とユニーク制約違反の処理をテストする
  - クリーンアップ機能の期限切れレコード削除と件数返却をテストする
  - Supabaseクライアントをモックして各サービス関数を検証する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 3.1, 3.2_

- [ ] 4.2 通知重複防止の統合テストを作成する
  - 同一通知を2回処理した場合、2回目がスキップされることを検証する
  - 繰り返しイベントの異なるオカレンスが独立して送信されることを検証する
  - 複数の通知設定（30分前、1時間前等）が独立して管理されることを検証する
  - センチネル通知にも重複防止が適用されることを検証する
  - DBチェック失敗時にフェイルオープンで送信が実行されることを検証する
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 4.1, 4.2, 4.3, 4.4_
