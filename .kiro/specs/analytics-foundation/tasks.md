# Implementation Plan

## Task 1: PostHog SDKの導入とプロバイダー設定

- [ ] 1. PostHog SDKの導入とプロバイダー設定
- [x] 1.1 PostHogパッケージのインストールと環境変数設定
  - `posthog-js` パッケージをプロジェクトに追加する
  - PostHog Cloudのプロジェクトを作成し、APIキーとホストURLを取得する
  - 環境変数 `NEXT_PUBLIC_POSTHOG_KEY` と `NEXT_PUBLIC_POSTHOG_HOST` を `.env.local` に設定する
  - `.env.example` にキー名のみ（値なし）を追加して、他の開発者が設定内容を把握できるようにする
  - _Requirements: 1.1, 1.3_

- [x] 1.2 PostHogProviderの作成とルートレイアウトへの統合
  - PostHog SDKをクライアントサイドで初期化するProviderコンポーネントを作成する
  - 環境変数からAPIキーとホストURLを読み込み、未設定の場合はSDKを初期化せずに警告ログを出力する
  - 開発環境ではデバッグモードを有効化し、本番環境ではデバッグを無効にする
  - Cookieを使用しない設定（メモリベースの永続化）で初期化し、Cookie同意が不要な構成にする
  - 自動ページビューキャプチャを無効化し、手動キャプチャコンポーネントに委譲する
  - SDK初期化をtry/catchで囲み、失敗時もアプリケーションの動作を妨げないようにする
  - ルートレイアウトにPostHogProviderを既存のプロバイダー（テーマ、ツールチップ）と並列で配置する
  - バンドルサイズへの影響を最小化するため、dynamic importで遅延読み込みする
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 6.2, 6.3, 6.4_
  - _Contracts: PostHogProvider State_

## Task 2: ページビュー自動トラッキングの実装

- [x] 2. ページビュー自動トラッキングの実装
  - Next.js App Routerのパス名とクエリパラメータの変更を監視するコンポーネントを作成する
  - パスが変更されるたびにPostHog SDKのページビューイベントをキャプチャする
  - SSR初期表示時とSPA遷移（クライアントサイドナビゲーション）の両方でページビューを記録する
  - URL、リファラー、タイムスタンプはSDKが自動付与するため、手動でのプロパティ設定は不要
  - Next.js 16のSuspense要件に対応し、適切な境界内でコンポーネントを使用する
  - PostHogProvider内の子コンポーネントとして配置する
  - 認証ルートと公開ルートの区別はPostHog Cloudのダッシュボードフィルタで対応する
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 5.1_

## Task 3: 型安全なカスタムイベント送信モジュールの作成

- [x] 3. (P) 型安全なカスタムイベント送信モジュールの作成
  - カスタムイベント名（event_created, event_updated, event_deleted, event_moved, event_resized, guild_switched, view_changed, calendar_navigated）の型定義を作成する
  - 各イベントに対応するプロパティの型マップを定義する（PIIを型レベルで排除）
  - イベント名とプロパティを受け取り、PostHog SDKのcapture APIを呼び出す型安全な送信関数を実装する
  - PostHog SDKが未初期化の場合は何もしない（no-op）グレースフルデグラデーションを実装する
  - PostHogインスタンスを取得するヘルパー関数を作成する
  - 送信関数の単体テストを作成する（正常系: SDKのcaptureが正しい引数で呼ばれること、異常系: SDK未初期化時にエラーが発生しないこと）
  - Task 2と並列実行可能（共有リソースなし、PostHog SDKのみに依存）
  - _Requirements: 3.6, 4.2, 6.1_
  - _Contracts: AnalyticsWrapper Service_

## Task 4: イベントCRUD操作のトラッキング統合

- [x] 4. イベントCRUD操作のトラッキング統合
- [x] 4.1 イベント作成・編集・削除のトラッキング
  - カレンダーコンテナの予定作成成功コールバックに `event_created` トラッキングを追加する（終日フラグ、色、通知有無をプロパティに含める）
  - 予定編集成功コールバックに `event_updated` トラッキングを追加する（変更されたフィールド名の配列をプロパティに含める）
  - 予定削除確認後の成功コールバックに `event_deleted` トラッキングを追加する
  - イベントタイトル、説明文などの個人を特定できる情報はプロパティに含めない
  - _Requirements: 3.1, 3.2, 3.3, 3.6_

- [x] 4.2 (P) ドラッグ操作のトラッキング
  - ドラッグ&ドロップによるイベント移動成功時に `event_moved` トラッキングを追加する
  - ドラッグリサイズによるイベント時間変更成功時に `event_resized` トラッキングを追加する
  - オプティミスティック更新が成功した場合（リバートされなかった場合）にのみトラッキングする
  - Task 4.1と並列実行可能（異なるイベントハンドラーを対象、ファイル競合なし）
  - _Requirements: 3.4, 3.5_

## Task 5: ギルド切替・ダッシュボード操作のトラッキング統合

- [x] 5. (P) ギルド切替・ダッシュボード操作のトラッキング統合
- [x] 5.1 (P) ギルド切替のトラッキング
  - ダッシュボードのギルド選択ハンドラーに `guild_switched` トラッキングを追加する
  - プロパティにギルドIDのみを含め、ギルド名やアイコンURLは含めない
  - Task 4と並列実行可能（異なるコンポーネントを対象）
  - _Requirements: 4.1, 4.2_

- [x] 5.2 (P) カレンダー表示切替・ナビゲーションのトラッキング
  - カレンダーの表示モード切替ハンドラーに `view_changed` トラッキングを追加する（切替先のビュータイプ: day/week/month をプロパティに含める）
  - カレンダーの期間移動ハンドラーに `calendar_navigated` トラッキングを追加する（移動方向: prev/next/today をプロパティに含める）
  - Task 5.1と並列実行可能（異なるコンポーネントを対象）
  - _Requirements: 5.2, 5.3_
