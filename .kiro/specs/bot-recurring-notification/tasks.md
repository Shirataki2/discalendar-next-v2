  - Web側のRRULE展開・バリデーション・要約テキスト生成・日付フォーマット関数を共有パッケージに移行する
  - ESM エントリポイントと型定義ファイル生成を設定する
  - TypeScript ビルド設定（ES2022ターゲット、declaration有効）を構成する
  - `rrule` パッケージ以外の外部依存を持たないことを確認する

- [x] 1.2 Web側のインポートを共有パッケージに切り替え、既存テストを移行する
  - Web側でRRULE関数を参照している箇所のインポートパスを共有パッケージに変更する
  - 既存のRRULE展開テストを共有パッケージに移動し、パスを修正する
  - Web側のビルド・テストが既存と同じ結果を返すことを確認する
  - EXDATE除外の動作がテストでカバーされていることを確認する

- [x] 2. Bot側の型定義とシリーズ取得サービス
- [x] 2.1 (P) Bot側にイベントシリーズのレコード型を追加する
  - event_series テーブルのカラム構造に対応する型を定義する
  - RRULE文字列、開始日時、持続時間（分）、除外日リスト、通知設定のフィールドを含める
  - 通知設定フィールドは既存の通知ペイロード型を再利用する
  - _Requirements: 6.1, 6.2_

- [x] 2.2 (P) event_series テーブルから全ギルドのシリーズを取得するサービス関数を実装する
  - Supabase経由でevent_seriesテーブルから有効な全シリーズレコードを取得する関数を追加する
  - 件数上限のパフォーマンスガードを設定する
  - 既存のResult型パターンとエラー分類（classifySupabaseError）に従う
  - Supabaseモックを使用したテストを作成し、正常取得・エラーハンドリングを検証する
  - _Requirements: 2.1, 2.2, 2.3, 2.4_

- [ ] 3. 通知タスクへの繰り返し予定通知の統合
- [ ] 3.1 シリーズオカレンスをイベントレコード互換に変換するヘルパーを実装する
  - シリーズレコードとオカレンス日時を入力として、既存の通知判定関数に渡せる形式に変換する
  - オカレンスの開始時刻と持続時間から終了時刻を計算する
  - 合成IDを生成し、シリーズの名前・説明・色・通知設定などのフィールドをマッピングする
  - 通常オカレンスと終日オカレンスの両方の変換をテストで検証する
  - _Requirements: 3.2, 6.3_

- [ ] 3.2 通知処理を拡張し、シリーズの取得・RRULE展開・通知判定を統合する
  - 単発予定の取得と並列でシリーズ取得を実行する
  - 各シリーズのRRULEを通知ウィンドウ（現在時刻〜最大7日先）の範囲で展開する
  - 展開された各オカレンスを変換ヘルパーで変換し、既存の通知判定・送信処理に流す
  - シリーズ取得失敗時はエラーログのみ出力し、単発予定の通知処理は継続する
  - 通知チャンネル設定の解決に、シリーズのギルドIDも含める
  - _Requirements: 3.1, 3.3, 3.4, 3.5, 5.1, 5.2, 5.3_

- [ ] 3.3 統合テストで後方互換性・EXDATE除外・終日通知を検証する
  - シリーズオカレンスの通知送信が正しく行われることをテストする
  - EXDATE付きシリーズで除外日のオカレンスが通知されないことを確認する
  - 終日シリーズオカレンスでJST午前0時基準の通知時刻計算が正しいことを検証する
  - シリーズ取得失敗時に単発予定の通知が影響を受けないことを確認する
  - 開始時刻一致時のセンチネル通知が正しく送信されることを検証する
  - 既存の単発予定通知テストが全て変更なく通過することを確認する
  - _Requirements: 3.4, 3.5, 4.1, 4.2, 5.1, 5.3_
