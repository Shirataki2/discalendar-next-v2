# Implementation Plan

- [ ] 1. ユーティリティ基盤の構築
- [x] 1.1 (P) 日時テキストパーサーの実装
  - テキスト形式の日時文字列（`YYYY/MM/DD HH:mm`, `MM/DD HH:mm`, `YYYY-MM-DD HH:mm`）をパースする関数を既存の日時ユーティリティに追加する
  - 年省略時は現在年をデフォルトとする
  - パース成功時は年・月・日・時・分のパーツを返し、失敗時はユーザー向けエラーメッセージを返すResult型を使用する
  - パース後に既存の日付バリデーション関数で論理的な妥当性（月末、うるう年等）を検証する
  - 正常系3フォーマット、年省略、無効入力、境界値（月末、うるう年）のユニットテストを作成する
  - _Requirements: 3.1, 3.2, 3.3, 3.4_

- [x] 1.2 (P) モーダルビルダーヘルパーの実装
  - イベント作成用モーダルを構築する関数を作成する（タイトル、説明、開始日時、終了日時、終日フラグの5フィールド）
  - タイトルは必須・Short形式、説明は任意・Paragraph形式、日時フィールドにはフォーマット例のプレースホルダーを設定する
  - 終日フラグは Short形式で `true` / `false` のテキスト入力とし、プレースホルダーで入力方法を説明する
  - イベント編集用モーダルを構築する関数を作成する（既存イベントの値をプリフィルで設定）
  - 編集モーダルの日時プリフィル値は既存のJSTフォーマット関数でテキスト変換する
  - モーダルの customId として作成用（`event-create`）と編集用（`event-edit:{eventId}`）のパターンを定数として定義し、customIdからイベントIDを抽出するヘルパー関数も作成する
  - 5つの ActionRow、正しい customId・ラベル・スタイル・プレースホルダー・プリフィル値のユニットテストを作成する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 1.6, 2.1, 2.2_

- [ ] 2. ModalSubmitハンドラの実装
- [ ] 2.1 モーダル送信処理の実装
  - ModalSubmitInteraction を受け取り、customId でイベント作成と編集をルーティングするハンドラ関数を作成する
  - 不明な customId の場合は処理をスキップしログ出力のみ行う
  - フィールド値を抽出し、タイトル必須チェック、日時パース、開始≧終了チェック、終日フラグ（true/false以外は不正）のバリデーションを順次実行する
  - バリデーションエラーはすべてエフェメラルメッセージで具体的なエラーメッセージを返す
  - restricted guild の場合は既存の権限チェック関数を使って管理権限を検証する
  - 作成の場合は既存のイベント作成サービスを呼び出し、編集の場合はcustomIdからイベントIDを抽出して既存の更新サービスを呼び出す
  - 成功時は既存のEmbed構築関数でイベント詳細を返信し、Supabaseエラー時はエフェメラルエラーメッセージを返す
  - 各バリデーションパターン（タイトル未入力、日時パースエラー、前後関係エラー、終日フラグエラー）、権限エラー、Supabase保存エラー、正常系のユニットテストを作成する
  - _Requirements: 4.1, 4.2, 4.3, 4.4, 4.5, 5.1, 5.2, 5.3, 6.1, 6.2, 6.3, 7.2, 7.3, 8.1, 8.2_

- [ ] 2.2 bot.ts への ModalSubmit ハンドラ登録
  - `interactionCreate` イベントハンドラに `isModalSubmit()` 判定を追加し、モーダル送信時に新しいハンドラ関数にディスパッチする
  - 既存の `isChatInputCommand()` 判定の前に ModalSubmit チェックを配置する
  - エラーハンドリングは既存のコマンドエラーハンドリングと同じパターン（try-catch + followUp）を適用する
  - _Requirements: 7.1_

- [ ] 3. コマンドへのモーダル統合
- [ ] 3.1 (P) `/create` コマンドにモーダル表示を追加
  - `/create` コマンドの実行関数に、スラッシュオプション（name）が未指定の場合にモーダルを表示する分岐を追加する
  - モーダル表示前に restricted guild の権限チェックを行い、権限不足の場合はエフェメラルエラーを返す
  - オプションが指定されている場合は既存のスラッシュオプション処理をそのまま実行し後方互換性を維持する
  - _Requirements: 1.1, 9.1_

- [ ] 3.2 (P) `/edit` コマンドにプリフィル付きモーダル表示を追加
  - `/edit` コマンドの実行関数に、`event` オプション以外の編集オプションが全て未指定の場合にモーダル表示パスに分岐するロジックを追加する
  - モーダルパスではイベント名で検索し、見つからなければエフェメラルエラーを返す。見つかれば既存値をプリフィルしたモーダルを表示する
  - モーダルパスでは `deferReply()` を呼ばず `showModal()` を直接呼び出す（Discord APIの制約: showModalは最初のレスポンスである必要がある）
  - 編集オプションが指定されている場合は既存の処理フローをそのまま実行し後方互換性を維持する
  - 指定イベントが存在しない場合のエフェメラルエラーメッセージを返す
  - _Requirements: 2.1, 2.2, 2.3, 9.2_

- [ ] 4. 結合テスト
- [ ] 4.1 全フローの結合テスト
  - `/create`（オプションなし）→ モーダル表示 → Submit → イベント作成 → Embed返信の一連のフローをテストする
  - `/edit event_name`（編集オプションなし）→ イベント検索 → モーダル表示（プリフィル確認）→ Submit → イベント更新 → Embed返信の一連のフローをテストする
  - `/create name:xxx start_year:2025 ...`（全オプション指定）で従来通りのフローが動作する後方互換性テストを作成する
  - restricted guild で権限なしユーザーがコマンド実行時にエフェメラルエラーが返されることを検証する
  - _Requirements: 1.1, 2.1, 2.2, 5.1, 5.2, 6.1, 6.2, 8.1, 8.2, 9.1, 9.2_
